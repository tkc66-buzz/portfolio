# Feature Specification: 目次（TOC）とセクション粒度の設計

**Feature Branch**: `001-portfolio-toc`  
**Created**: 2025-12-24  
**Status**: Draft  
**Input**: User description: "まずは目次のコンテンツを考えたい。ファイルでいうと @src/app/page.tsx なのかな。コンテンツ自体はこれでいいのかな？もっといい粒度にできたりする？"

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 初見で人物像がつかめる目次と構成 (Priority: P1)

初めて来た訪問者が、ページ上部の目次（TOC）と見出し構成を見るだけで「この人は何者で、
何をやってきて、何が得意か」を短時間で把握できる。

**Why this priority**: ポートフォリオの価値は最初の数十秒で決まるため、情報設計が最優先。

**Independent Test**: 目次だけを見て、主要情報（人物像/実績/連絡先）に到達できるかを確認できる。

**Acceptance Scenarios**:

1. **Given** 初見ユーザーがページを開いた状態、**When** 目次を見る、**Then** 主要セクションが一覧で分かり、どこに何があるか理解できる
2. **Given** 初見ユーザーがページを開いた状態、**When** 目次から任意のセクションを選ぶ、**Then** 目的の内容に迷わず到達できる

---

### User Story 2 - 採用/協業視点で実績を探しやすい (Priority: P2)

採用担当や協業者が「実績」「役割」「成果」「使える技術」「強み」を効率よく拾えるように、
セクションの粒度が “探す行動” に合っている。

**Why this priority**: ポートフォリオは検索・比較されるため、情報の発見可能性が重要。

**Independent Test**: 目次から「実績（Projects）」に飛び、各プロジェクトの要点が不足なく揃うことを確認できる。

**Acceptance Scenarios**:

1. **Given** 採用/協業者がプロジェクト詳細を探している、**When** 目次から Projects に移動する、**Then** 各プロジェクトに「一言概要/役割/技術/成果または学び/リンク」が見つかる

---

### User Story 3 - 将来の更新がしやすい粒度ルール (Priority: P3)

サイトオーナーが後から経歴や実績を追加する際、どのセクションに何を書けばよいか迷わない
（増築に強い情報設計）。

**Why this priority**: ポートフォリオは更新が前提。運用しづらい構成はすぐ陳腐化する。

**Independent Test**: “新しい実績1件” を足すときの置き場所が一意に決まるかを確認できる。

**Acceptance Scenarios**:

1. **Given** 新しい実績を追加したい、**When** ルールに従って配置先を判断する、**Then** Projects（または Experience）など適切な場所が迷いなく決まる

---

### Edge Cases

- 目次の項目が多すぎて、逆に探しづらくなる（8項目を超える、階層が深い）
- モバイルで目次が押しづらい/読みにくい（小さい画面でも操作できる必要）
- セクションの内容がまだ未確定で “空” に近い場合（「準備中」などで誤解を生まない表現が必要）
- プロジェクトの情報が欠けている（成果が数値で出せない等）場合の扱い

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: サイトはページ上部に **目次（TOC）** を表示し、主要セクションの一覧を示さなければならない
- **FR-002**: ユーザーは目次から **任意の主要セクションへ移動** できなければならない
- **FR-003**: 目次に含める主要セクションは、最低限「Profile（短い自己紹介）/ Experience（経歴・活動）/ Projects / Skills / Contact」を含まなければならない
- **FR-004**: 各セクションは「見出し名だけで中身が推測できる」命名でなければならない（例: “About” だけで曖昧にしない）
- **FR-005**: Projects は各項目に最低限「一言概要 / 役割 / 技術（カテゴリ） / 成果または学び / 外部リンク（任意）」を含まなければならない
- **FR-006**: 目次の項目数は原則 **8項目以下** に保たなければならない（超える場合は統合または階層化の理由を明記する）

### Key Entities _(include if feature involves data)_

- **Section**: ポートフォリオの情報を分類する単位（例: Profile, Experience, Projects, Skills, Contact）
- **TOC Item**: Section への導線（表示名、並び順、対象セクション）

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 目次から主要セクション（FR-003）へ到達でき、リンク切れや重複がない
- **SC-002**: 目次の項目数が 8 以下で、ページ全体をスクロールせずに一覧として把握できる
- **SC-003**: 初見の第三者が 2 分以内に「人物像」「代表実績」「連絡先」を見つけられる（簡易ユーザーテストで検証可能）
- **SC-004**: モバイル表示でも目次と各セクションが読みやすく、操作に支障がない（横スクロール前提にならない）

## Assumptions

- この機能の対象は「情報設計（目次・見出し・粒度ルール）」であり、詳細なコピー（文章そのもの）は
  次フェーズで差し替える前提とする
- 現状の大枠（Profile/Experience/Projects/Skills/Contact）は維持しつつ、命名と役割定義で分かりやすくする
- “レトロ” はビジュアルの方針であり、読みやすさ・到達しやすさを犠牲にしない（憲法準拠）

## Proposed TOC (information architecture)

目次は「見出し名だけで中身が推測できる」ことを優先し、以下を基本形とする。

- **Profile**: 何者か（肩書き/強み/今やっていること）を 3〜5 行で提示
- **Experience**: 経歴・活動の流れ（役割の変遷、転機、現在に至る理由）
- **Projects**: 代表実績（証拠）。各プロジェクトは要点セット（FR-005）で揃える
- **Skills**: 得意領域・スタック・仕事の進め方（できることの棚卸し）
- **Contact**: 連絡導線（SNS/メール/フォーム等）と「どう連絡してほしいか」

※ 項目を増やす場合は FR-006（8項目以下）を守り、増やす代わりに統合候補を検討する。

## Non-Goals

- デザインテーマ（レトロ表現）の刷新や大幅な UI リニューアルは扱わない
- CMS 導入、ブログ機能、検索機能、認証など“目次設計”を超える機能追加は扱わない
- 各セクションの最終コピー（文章・具体実績の全入力）をこの機能の完了条件にしない
